## Problem Statement

在 Mac Studio (Apple M1 Max, 32GB) 上搭一套 **100% 本地、离线、免费、开源** 的英语口语对话练习助手。
目标是练 **自然连贯的日常对话**（不是补词汇量——用户词汇量已足够，约 2000 句分两个 Anki deck）。
核心诉求：**最低端到端轮次延迟**（用户说完 → 助手开口），**不打断、能一直聊下去**，并把用户
的 Anki 句子织进对话里做"对话式复习"。

## What Makes This Cool

把被动的 Anki 句子复习（只能"看")变成 **主动的对话式间隔复习（conversational spaced
repetition）**：助手围绕用户正在背的高频句子，自然地引导日常对话，让这些句子在真实语境里被
反复用到。这是这个项目区别于"又一个本地语音助手"的真正价值点。

## Constraints

- 完全本地 / 离线 / 免费 / 开源，无任何云 API。
- 硬件：Mac Studio，Apple M1 Max，32GB 统一内存（机器上还跑着其它 app，要平衡资源）。
- 已装：Ollama (`qwen3.5:4b`, `qwen3.5:9b`, `deepseek-r1:8b`, `nomic-embed-text`, `bce-reranker`)、
  MLX 框架（"Rapid MLX"，需升级到最新）、uv 0.8.0、Homebrew、Python 3.13。
- 延迟优先于吞吐；TTFT（首字延迟）是关键指标。
- 不要过度复杂，满足需求即可。

## Premises（已与用户确认）

1. 个人工具，先跑通低延迟闭环，再谈打磨。**[已确认]**
2. 纯 ASR→LLM→TTS 闭环 **无法直接评判发音**——Whisper 会把语音"纠正"成正确文本，用户的
   语法/发音错误模型看不到。"更像 native" 的现实机制是：① 用户模仿助手的地道措辞与 TTS 发音；
   ② 助手用更地道的说法复述/接话。用户选择 **不显式纠错的自然对话**，正好与此一致。**[已确认]**
3. 架构走 **Fork 现成仓库改**，不重复造轮子；用 GPU/MLX 但平衡资源、别太复杂。**[已确认]**

## Cross-Model Perspective（三方独立调研共识）

来源：本会话的网络调研 + Codex 高强度冷读 + Claude 独立子代理。三方在以下选型上**高度一致**：

| 组件 | 共识选型 | 理由 |
|---|---|---|
| TTS | **Kokoro-82M** | 三方零异议。Apache-2.0，RTF≈0.03，MOS 4.2，对话够自然 |
| LLM | **`qwen3.5:4b`**（非 9b） | 4B TTFT≈350ms vs 9B≈650ms；延迟感由 TTFT 主导 |
| VAD | **Silero VAD** | 端点检测，亚秒级收尾 |
| 编排 | 自写 ~150-250 行 asyncio（理想）；本项目改为 **Fork 参考仓库** | 框架对单机个人工具过度封装 |

**分歧（已决议）：**
- ASR：Codex 首选 Moonshine（英语专用、流式、MIT、CPU/ONNX 不抢 GPU）；参考仓库用 faster-whisper
  (CPU)。→ **采用 faster-whisper `small.en` (CPU)**（仓库已集成、英语、CPU 跑不抢 GPU，符合"平衡资源"），
  Moonshine 作为后续延迟升级备选。
- TTS 后端：kokoro-onnx (CPU) vs mlx-audio (GPU)。→ **采用 mlx-audio Kokoro (GPU)**（仓库默认、用户
  接受用 GPU），但知悉其与 LLM 的 GPU 争用（见 Risks）。

**端点检测黄金参数（Codex）：** 语音起始 60-90ms 触发；静音 **250-350ms** 收尾（比默认放宽，
照顾"不打断、让用户把话说完"的诉求）；pre-roll 150-200ms；tail 200ms；单句硬上限放宽到 ~8s
（允许长句）；**按静音收尾，不按模型标点收尾**。

## Approaches Considered

### Approach A: 自写极简 asyncio 闭环（最低延迟，未选）
Moonshine(CPU) + qwen3.5:4b + kokoro-onnx(CPU) + Silero + ~200 行 asyncio。延迟最低、GPU 全给 LLM，
但要自己处理音频线程/采样率，多花约一天。**未选——用户要求不重复造轮子。**

### Approach B: Fork 参考仓库裁剪（已选）✅
Fork `eauchs/speech-to-speech-pipeline`（Apple Silicon/MLX、已自带 barge-in）：
faster-whisper(CPU) + OpenAI 兼容 LLM(Ollama) + mlx-audio Kokoro(GPU)。今天就能跑通，省掉从零搭架构，
现成的 asyncio + 打断逻辑可直接用。GPU 分工天然平衡（ASR 在 CPU，LLM+TTS 用 GPU）。

### Approach C: 开箱 WebUI（最快跑起来，未选）
open-webui 语音模式 + Ollama。零脚本但延迟最高且不可控，与"最低延迟"冲突。**未选。**

## Recommended Approach（= Approach B + Rehearse 定制）

**基础：** Fork `eauchs/speech-to-speech-pipeline`。

**资源分工（关键，规避头号盲区 GPU 争用）：**
- ASR → **faster-whisper `small.en`，CPU/int8**（不抢 GPU）
- LLM → **Ollama `qwen3.5:4b`，GPU**，常驻保温，**非思考/instruct 模式**
- TTS → **mlx-audio Kokoro，GPU**（与 LLM 在 GPU 上分时；靠句级流式 + 短回复缓解争用）

**针对 Rehearse 的四处定制改造：**

1. **连续不打断的对话循环**：去掉/弱化"自动退出"，循环常驻直到用户显式说退出（如 "stop" / Ctrl-C）。
   保留 barge-in（用户可随时插话打断助手，这本身就自然），但 VAD 收尾放宽到 250-350ms，
   不要把用户的话切断。

2. **Anki 句子注入（Rehearse 的灵魂）**：
   - 用户从 Anki 导出两个 deck 为 `.txt`（Notes in Plain Text，tab 分隔）→ 脚本解析出句子列表。
   - **MVP（Phase 1）会话播种**：每次会话按"最久未练"顺序抽 ~10-15 句，写进 system prompt：
     "自然地把这些句子/句型织进日常对话，引导用户在真实语境里用到它们；不要生硬罗列。"
   - **增强（Phase 2）语义检索**：用已装的 `nomic-embed-text` 把全部句子向量化存本地（用户偏好 SQLite，
     可用 sqlite-vec 或纯文件），对话进行中按当前话题检索语义相关句子、用 `bce-reranker` 重排，
     动态喂给助手 → 实现"找相关对话帮我复习"。同时记录哪些句子已在对话中用过 → 间隔复习去重。

3. **System prompt（自然对话伙伴）**：日常高频口语；**不显式纠语法**；用更地道的母语者说法接话/复述；
   主动追问让对话延续；**plain speech only**（剥离 markdown/emoji，否则 TTS 会念出符号）；
   把当轮目标句子自然带入。

4. **English-only + 保温 + 非思考**：`small.en` ASR；`OLLAMA_KEEP_ALIVE=-1` 常驻；
   确认 `qwen3.5:4b` 跑非思考模式（避免 reasoning token 把 TTFT 拉爆）。**排除 `deepseek-r1:8b`**（思考模型不适合语音）。

## Open Questions

- Anki 导出格式：用户两个 deck 的字段结构？（句子在第几个 field，是否含中文翻译/提示）——影响解析脚本。
- 是否要"本轮练了哪些句子"的轻量进度回显（文本/日志），还是纯语音零界面？
- `qwen3.5:4b` 在 Ollama 里是否有 `/no_think` 或等价开关确认非思考模式。

## Success Criteria

- 端到端轮次延迟（用户说完→助手开口）目标 **< 1.5s**（暖机后），可接受 < 2.5s。
- 能连续多轮对话不被系统打断，直到用户主动退出。
- 一次会话中，目标 Anki 句子里至少有一部分被自然地带进对话。
- 全程离线，断网可用。

## Distribution Plan

个人自用，无需分发。交付物 = 本地 git 仓库 + `uv` 虚拟环境 + 一条启动命令（如 `uv run rehearse`）。
无 CI/CD 需求。

4. 改对话循环为"常驻不退出 + 放宽 VAD 收尾"，保留 barge-in。

## Risks / Reviewer Concerns（盲区清单）

1. **GPU/统一内存争用（头号盲区）**：LLM 与 mlx-audio TTS 都在 GPU；32GB 上叠加其它 app，尾延迟可能涨
   20-40%。缓解：句级流式 TTS（先念第 1 句，LLM 继续生成）、短回复、必要时把 TTS 切到 kokoro-onnx(CPU)。
2. **采样率不匹配 / 音频抖动（Codex 头号坑）**：设备 I/O 固定 48kHz，一次性重采样到 16kHz mono 喂 ASR，
   TTS 输出再采样回放，**绝不每轮重开音频设备**。否则 VAD 乱、爆音、感觉更慢。
3. **冷启动**：首次推理 30-60s；`OLLAMA_KEEP_ALIVE=-1` 保温。
4. **思考模式税**：必须确认 `qwen3.5:4b` 非思考模式，否则 TTFT 爆炸。
5. **发音无法被评判**（见 Premise 2）：这是产品的固有边界，用户已知悉并接受。
6. **Python 版本**（已更正）：`kokoro-onnx` 现支持 Python <3.14，**3.13 可直接用**；不必锁 3.12。
7. **Fork 不可信**（Codex 更正）：`eauchs/speech-to-speech-pipeline` 仅 3 commits / 7 stars / 无 release，是 demo
   不是地基。当"可挖的代码"，先 spike 再决定 fork 还是自写。

## v2 Hardening — Codex 对抗审查后修订（2026-05-27）

Codex（high→medium reasoning）对 v1 打分 **4/10，"overconfident"**。以下为已采纳的修订，**本节为权威 delta**，
与上文冲突处以本节为准。3 个 BLOCKER 全部解决。

### 用户硬件（决定双工模式）
- 输入：**DJI Mic Mini**（近场无线领夹麦，SNR 高，天然帮 VAD 区分人声与音箱漏音）
- 输出：**Mac Studio 内置音箱** 或 **AirPods Pro**

### BLOCKER 1 — 声学回声（v1 最大遗漏）→ 已解
外放时麦克风会拾取助手自己的 TTS → 自我转录、误触发 VAD、"自己打断自己"。barge-in 不解决回声。
**决策：v1 默认半双工**——TTS 播放期间静音麦克风捕获，播放结束 +150-200ms guard 再解除静音。
- 对 Mac Studio 音箱场景：安全，根除回声。
- 对 AirPods Pro（入耳，零漏音）：无害；可加 `--full-duplex` 开关启用插话打断。
- 代价：半双工下助手说话时不能插话（要等它说完）。助手回复保持简短即可，不影响"连续多轮对话"。

### BLOCKER 2 — GPU LLM+TTS 并发争用 → 已解
v1 删除"LLM 与 TTS 在 GPU 上重叠"。**决策：串行化**——TTS 保留在 GPU(mlx-audio)，但与 LLM 错开：
生成够一个可朗读句子 → 暂停 LLM 解码 → 跑 TTS 出声 → 恢复生成。既贴合"尽量用 GPU"，又消除同时抢 GPU
导致的 TTFT 抖动。句间可能有极小停顿，个人工具可接受。（备选：TTS 移到 kokoro-onnx/CPU。）

### BLOCKER 3 — 非思考模式是结构性的 → 已解
不能只靠 prompt 文本。**决策：走 Ollama 原生 `/api/chat` 的 `think: false`（或 Qwen `enable_thinking=False`），
并加自动探针**：发一条固定 probe，断言响应无 `<think>` 块且 TTFT 低于阈值；不达标则启动失败/告警。
若上游 OpenAI-compat 客户端无法干净传递该参数，就绕过它直接调 Ollama 原生 API。

### HIGH — 延迟目标不实 → 已修正
现实暖机中位数约 **1.4-2.0s，尾部 >2s**，瓶颈是 **LLM TTFT**（旧文 ≈350ms 无依据）。
**修正 Success Criteria：** `<1.5s` 降级为冲刺目标，不作验收标准。验收 = **第一天就埋 4 个时间戳**
（`vad_end → asr_done → llm_first_token → tts_first_audio`）并出 p50/p95。若暖机中位数 >1.6s，
**先砍回复长度/上下文，而不是先换 ASR**；真要换 ASR 用流式 CPU 的 Moonshine（whisper.cpp Metal 会重新引入 GPU 争用）。

### HIGH — Anki 注入太弱 → 已收紧
- 每轮只喂 **2-4 个目标句**（不是 10-15），降低 4B 模型忽略/生硬复读的概率。
- "练过"必须可度量，定义为三者之一：① 用户复述了目标；② 语义改写了目标；③ 之后无提示自发产出。
  每轮结构化记录"尝试了哪个目标 / 命中哪种"。**测不出就是表演。**
- **Phase 2 语义检索（nomic-embed + bce-reranker）推迟**：2000 句规模属过度工程，先让 MVP 证明
  "注入确实改变了对话行为"再做。

### HIGH — 上游不可信 / Python 已更正
- `eauchs/...` 当 demo 不当地基（见 Risks #7）。**先 Spike 验证 4 件事**，再决定 fork 还是自写 ~200 行：
  ① 连续循环不退出；② 非思考 qwen 且 TTFT 低；③ TTS 首音可接受；④ **半双工无回声**。
- `kokoro-onnx` 现支持 Python <3.14 → **3.13 直接用**，不锁 3.12（见 Risks #6）。

### 加固后的验收口径（取代旧 Success Criteria 第 1 条）
- 出具 p50/p95 四段延迟报告；目标 p50 ≤ ~1.6s、p95 ≤ 2.5s（达不到就调，不假装达标）。
- 半双工连续多轮，直到用户主动退出；AirPods 下可选全双工打断。
- 每轮记录目标句"尝试/命中"，会话末给一行复习小结。
- 全程离线。

### Codex 给的 Top 3（开工前必做）
1. 用真实 M1 Max 基准 harness 取代模糊延迟目标（埋四段时间戳）。
2. 开工前锁死双工策略（本节已锁：半双工默认 + AirPods 可选全双工）。
3. v1 去掉 GPU 上 LLM+TTS 并发（本节已锁：串行化）。

## v3 — Eng Review Lock（/plan-eng-review, 2026-05-27）

实现层决策锁定（本节为实现权威）：

- **D1 串行化实现 = 全串行 + 短回复上限**：系统提示强制回复 1-3 句；拿到**完整**回复后再逐句 TTS。
  不做 token 流式 overlap（Ollama 无法真正中途暂停；短回复下 overlap 仅省 ~0.3s）。GPU 零并发。
- **D2 打断 = 键盘中断 + AirPods 语音打断**：半双工默认无语音打断，助手说话时按键即截停；
  `--full-duplex`（AirPods）启用语音 barge-in。解决 v2 "保留 barge-in" 与半双工的自相矛盾。
- **D3 "练过"度量 = nomic 语义相似度（MVP）**：每轮用户话结束后，算 transcript 与各活跃目标句的 cosine
  相似度，超阈值记"复述/改写命中"。复用已装 nomic-embed，~20 行。"无提示自发产出"归 Phase 2。
- **D4 测试 = WAV fixture 集成 + 确定性模块全单测 + 手动全闭环**：真实本地模型集成测（按用户 real-tests 偏好）。

实现要求（非岔路，直接执行）：
- **模型保温**：Kokoro / faster-whisper / Ollama 各加载一次，绝不每轮重载（每轮重载慢 ~30x）。
- **单音频设备**：启动开一次 48kHz；一次性重采样 48k→16k 喂 ASR、Kokoro 24k 输出重采样回放；绝不每轮重开设备。
- **think:false 探针**：启动 health-check，断言响应无 `<think>` 块且 TTFT < 阈值，否则 fail-fast。

### NOT in scope（v1）
- 语义检索注入（Phase 2, nomic+bce-reranker）—— 2000 句规模属过度工程，先证明注入有效。→ TODOS T-P2-1
- 跨会话"无提示自发产出"追踪 + SQLite 间隔复习排程。→ TODOS T-P2-2
- 全双工外放 + 真 WebRTC AEC —— D2 用键盘中断 + AirPods 绕开。→ TODOS T-P2-3
- token 流式 TTS overlap —— D1 选全串行。
- 分发/打包/CI —— 个人自用。
- 发音/语法主动纠错 —— 产品固有边界（Premise 2），不显式纠错。

## v4 — Live-feedback tuning (2026-05-27)

After real-mic testing (DJI Mic Mini + Mac speaker, non-native speaker):
- **Endpointing made patient (deliberate, not drift):** auto-mode `end_silence_ms`
  300 → **1000ms**, `max_utterance_ms` 8s → **20s**. 300ms cut the user off mid-thought.
  Tunable via `--end-silence-ms`. Pinned by `test_default_endpoint_is_patient`.
- **`--manual-turns` mode:** press Enter to start/stop a turn — zero VAD time pressure.
  Best for a thinking non-native speaker; `is_stop()` + Ctrl-C exit.
- **ASR accuracy:** `vad_filter=True` (strip internal silence/noise → less hallucination
  on close-mic+speaker), `beam_size=5`, `condition_on_previous_text=False`; empty-result
  retry without vad_filter (don't lose quiet answers). `--asr-model` to upgrade to
  medium.en / distil-large-v3.
- **D3 off critical path:** `respond(embed=None)` skips scoring; tracking latches OFF
  after a runtime embed failure so it never blocks a turn.
- **`--debug`:** saves each turn's WAV + transcript to `debug/` (gitignored).
- **Confirmed via data:** thinking is OFF (TTFT ~0.3s, no think block). The ~3.3s
  record→response on a 7s utterance is asr(1.2)+llm-gen(1.7)+tts(0.6), all serial (D1),
  NOT thinking. Next latency lever (deferred): sentence-streamed TTS playback.
