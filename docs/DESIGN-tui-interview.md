## Problem Statement

Two additions to the working v1 voice assistant:

**F1 — TUI wrapper.** The user runs in Ghostty (macOS) daily and wants an interactive
launcher menu + live status instead of bare CLI prompts. Voice stays the main
interaction; the TUI is a launcher (pick mode / content source / options) + a passive
status surface (listening / thinking / speaking, final transcript, coverage, latency).

**F2 — Markdown-driven interview-prep mode.** The user is job-hunting (ESL) and already
generates interview-prep Markdown via ChatGPT/Claude (resume bullets; interview Q&A with
key points + verbatim scripts). They want LocalVocal to load a Markdown file by absolute
path, have the assistant act as an **interviewer/coach**, drive the conversation around
the doc's key points, let them recall & elaborate each bullet **from memory**, and track
which points were covered — so they can speak confidently in the real interview
(memorize key points, not the whole script).

Sample docs (very different shapes):
- `…/portfolio2/resume/ai-engineer.md` — `##` sections → `**bold**` job/project headers → `-` bullets (75 lines)
- `…/JobPilot/NeoFinancial/CTO-call-prep.md` — `Q1 —` questions, `-` answer bullets, `>` verbatim scripts, one-page call sheet, glossary/meta sections (387 lines / 33KB)

## What Makes This Cool

Turns a static prep doc into a **spoken recall drill**: the assistant interviews you on
your own bullets, you answer from memory, and it scores *coverage* (did you actually hit
the key points + the hard facts) instead of letting you ramble. Same idea as the Anki
"conversational spaced repetition" but for interview answers. The voice stack (ASR/LLM/
TTS/VAD/duplex) is fully reused; F2 is a new content source + persona + scorer, not a new app.

## Constraints

- Local / offline / free; reuse the existing v1 speech stack unchanged.
- Personal tool, macOS, Ghostty. "别太复杂" — satisfy the need, avoid over-engineering.
- F1 and F2 must be able to ship independently.

## Premises (confirmed by the joint review)

1. **F1 is lightweight, not Textual.** prompt_toolkit/questionary launcher + Rich `Live`
   status panel. Full Textual is over-engineering for a voice-first personal tool; revisit
   only if it grows into a real app (settings panes, history, device picker, charts).
2. **F1's real problem is ownership, not widgets.** Today `main_loop` owns the sounddevice
   streams, `audio_q`, the stdin interrupt watcher, and `input()`. You cannot wrap that in a
   TUI without races / blocked stdin. **Prerequisite: extract a headless `SessionEngine`.**
3. **F2 maps onto the existing content-source/persona/scorer architecture** — do NOT fork
   the speech pipeline. Generalize `Sentence` → a `PracticeItem` layer (compose, don't bloat).
4. **Do NOT blob the whole Markdown into the prompt.** The CTO doc is full of scripts, meta,
   glossary, adversarial notes — blob-context makes the assistant coach *on the document*
   instead of drilling recall. Parse once, embed items once, drill ONE item per turn.
5. **Coverage scoring must not be "semantic vibes."** Naive cosine over broad bullets tells
   flattering lies. Score the current item only, against the cumulative answer, bullet-level,
   plus **anchor checks** (numbers / named entities / acronyms / dates must be present).

## Cross-Model Perspective (joint review — Codex high-reasoning + Claude subagent)

Both independently converged. Key points (lightly paraphrased):

**F1 (both):** lightweight wrapper. Architecture:
- Extract `SessionEngine` from `run_loop`/`run_manual`. Engine thread owns ALL audio
  primitives (InputStream/OutputStream, HalfDuplexGate, audio_q, interrupt, VAD, ASR/LLM/TTS).
- UI owns ALL keyboard. No `input()` in the engine, no stdin thread inside it.
- UI → engine over `cmd_q`: start_auto / start_manual / begin_record / end_record /
  interrupt_reply / stop / shutdown / set_option.
- Engine → UI over `event_q` (immutable events): state=listening/thinking/speaking/idle,
  user_transcript_final, assistant_text, coverage_update, latency_update, error.
- PortAudio callback does ONE thing: copy audio blocks into the engine queue.
- **No true partial transcript in v1** — ASR is post-endpoint, not streaming. Show
  VAD/listening state + final transcript. (Partials later = interim ASR off a copied
  rolling buffer on a separate worker; never touch the main turn loop.)
- Pitfall: stdin/audio ownership leaking from the loop into the UI layer.

**F2 (both):** generic `PracticeItem`, structured parse, per-item drilling.
- `PracticeItem(id, kind, prompt, expected_points, support_snippets, source_title, section, tags, metadata)`.
  Adapt `Sentence → PracticeItem(kind="sentence", prompt=text, expected_points=[text, native?])`
  so Anki/English keeps working unchanged.
- Structured Markdown parse, two passes: (1) line-oriented block parser (headings, bullets,
  blockquotes, bold labels); (2) extractor profile per doc shape:
  - resume: each bold project/job header → one item; bullets → expected_points; heading → section.
  - interview Q&A: `Q1/Q2/OPEN/CLOSE/REVERSE Qs` → items; bullets → expected_points;
    blockquotes → support_snippets / verbatim_script; glossary / "why this works" /
    adversarial-review sections → coach-only notes or ignored (NOT drill targets).
- Drill a 33KB doc: parse once, embed each item's retrieval text once with `nomic-embed-text`.
  Session state: `current_item_id`, `covered_points`, `attempt_count`. Each turn prompt =
  persona + current question + MISSING bullets as hidden guidance + maybe 1-2 support snippets.
  Advance by agenda order or least-covered-first; use retrieval only as a fallback to pick the
  next item / surface a snippet — never re-retrieve the whole doc every turn.
- Coverage scoring: current item only; against the CUMULATIVE answer for that question;
  bullet-level (one embedding sim per expected bullet); **plus anchor checks for factual
  bullets** (numbers, entities, acronyms, dates). `covered if sim >= calibrated_threshold AND
  required anchors present`. Recalibrate — do NOT reuse the 0.50 sentence threshold blindly.
- Persona: "interviewer first, coach second." One question at a time. User answers from
  memory. Weak coverage → short probing follow-up. Reveal hints/scripts only when the user
  stalls or asks. Keep the plain-spoken-English constraints from `prompt_builder`.
- Pitfall: treating coverage as semantic vibes → fake "covered" scores.

**Build order (Codex):** 1) extract SessionEngine, 2) PracticeItem + adapters, 3) ship F2 on
CLI (`--content markdown --path /abs --mode interview`) — higher value, lower risk, 4) add
lightweight TUI on engine events, 5) defer Textual. F1 and F2 ship independently; the only
hard shared prerequisite is the engine extraction (needed for a clean TUI).

## Approaches Considered

### F1 — TUI
- **A. Lightweight (chosen):** prompt_toolkit launcher + Rich Live status over a `SessionEngine`
  + cmd/event queues. ~100-150 lines of UI. Effort M, risk Low.
- **B. Full Textual app:** real panes/history/device-picker. Effort L, risk Med (async model vs
  the threaded audio loop). Deferred until it's actually app-shaped.
- **C. No TUI, just prettier CLI (Rich prints):** Effort S. Rejected — user explicitly wants menus.

### F2 — Interview-prep
- **A. PracticeItem + structured dual-path parser + per-item interview loop (chosen):** reuse
  pipeline; new content source + persona + coverage scorer. Effort M-L, risk Med (parser edge
  cases). Highest value.
- **B. Blob whole doc into system prompt, let LLM drive:** Effort S. Rejected — coaches on the
  doc, not recall; blows context; no real coverage metric.
- **C. Flat key-point extraction (no Q→bullet structure):** Effort S-M. Rejected — loses the
  interviewer "ask the question, check the answer" loop that makes it a drill.

## Recommended Approach

Reuse the v1 speech stack untouched. Add, in this order:

1. **`PracticeItem`** type + `Sentence→PracticeItem` adapter (Anki path unchanged).
   `prompt_builder`, `session_seeder`, scorer all operate on `PracticeItem`.
2. **`markdown_loader.py`** — block parser + per-shape extractor (resume / Q&A); skip meta
   sections; pre-embed items with nomic on load.
3. **`interview_coach`** persona prompt + **`CoverageTracker`** (bullet-level cosine + anchor
   checks, current-item + cumulative answer, recalibrated threshold).
4. **Ship F2 on the existing CLI:** `localvocal --content markdown --path /abs/file.md --mode interview`.
5. **`SessionEngine`** extraction (cmd_q/event_q; engine owns audio, UI owns keyboard).
6. **`launcher.py` + `status_display.py`** lightweight TUI on engine events (`--tui` / interactive default).

## Open Questions / Decisions for you

  no refactor needed), OR do the SessionEngine extraction first (Codex's clean-refactor
  preference) then F2 + TUI together. Value says F2 first; cleanliness says engine first.

- **More real samples:** the two given docs are unusually clean. Need 1-2 more of your actual
  ChatGPT-generated prep docs to harden the parser (Codex + subagent both flag messy real docs).

## Risks / Pitfalls (from the joint review)

1. **F1 ownership leak (top F1 risk):** layering a TUI before removing `input()`/stdin
   ownership from the loop → races, blocked stdin, half-dead sessions. Extract the engine first.
2. **F2 fake coverage (top F2 risk):** cosine-only scoring over broad bullets over-credits
   vague talk. Enforce anchor checks (facts must appear) + score current-item/cumulative.
3. **Messy real prep docs:** ChatGPT output has inconsistent Q labels, scripts buried in
   paragraphs, partial-thought bullets. Build a load-time validator (warn on <N items / items
   with 0 bullets) + a fallback that shows the parsed structure for the user to confirm. Test
   on real samples before trusting it.
4. **Ctrl-C mid-turn:** flush stats/coverage to a shared dict before each await/blocking step
   so the summary survives an interrupt during LLM/TTS.
